# UX Decisions

Append-only log of UX architecture decisions. Each entry: date, decision, rationale, evidence/status.

## 2026-07-02 — Period navigation on the Velocity Cockpit (temporal nav ≠ a forbidden control)

- **Decision:** Add a **prev/next period stepper** to the Productivity Stats cockpit (`‹ label ›` pill left of the `TimeRangeSwitch`) so the user can browse **past** days/weeks/months, not just the current rolling period. Mechanism: `computeProductivityStats(events, range, nowMs, offset)` gained an `offset` param — it shifts *only* the period window anchor back by `offset` whole periods (`day`=24h, `week`=7d, `month`=30d), while **lifetime views stay anchored to the real `nowMs`** (heatmap, streaks, all-time counters, and the rolling-average "beat your usual" red zone are NOT shifted — they describe *now*, not the browsed period). Offset is **ephemeral** (component state, not persisted; the range choice remains the only persisted stats pref) and **resets to 0 on range change** (offsets aren't comparable across granularities). The stepper is **hidden for range `all`** (stepping is meaningless). Labels are relative ("Today"/"Yesterday"/"{n} days ago", …) with the exact date range in the tooltip; the center label click jumps back to the current period. Arrow gating: `‹` disabled when no data exists before the shown window (`canGoOlder`), `›` disabled at the present (`canGoNewer`). When browsing a past period, the momentum trend suffix falls back to a neutral "vs prev." and the idle headline reads "Nothing shipped in this period" instead of the present-tense CTA.
- **Rationale:** The manifest guardrail says the cockpit is read-only — "never add a control, a filter beyond the existing time-range switch" (§1.1, `stats_dashboard.forbidden`). This resolves the apparent tension: **period navigation is an extension of the time-range switch on the *same* (time) axis, not a new control class.** The guardrail's real intent — verified against `forbidden: [mutation, task_lifecycle_action, durable_configuration, global_header_button]` — is to bar *operational* controls: mutations, durable config, and **data filters that add a new slicing dimension** (project/agent/label). Temporal navigation adds no new dimension, persists nothing, and mutates nothing; time is the one axis the cockpit already governs, so navigating along it is in-spirit while filtering across new ones stays banned. Rejected: a free-form date picker / calendar (turns the celebration surface into an operator console, blows the complexity+honesty budget); a second nav row under the header (duplicates the time axis the range switch already owns, extra chrome). Reworded the §1.1 guardrail + `stats_dashboard.allowed/forbidden` so the distinction (temporal-nav allowed, data-filter forbidden) is explicit and future agents don't re-litigate it.
- **Evidence/status:** `Observed` — `src/mainview/utils/productivityStats.ts` (`offset` param, `periodSpanMs`, `canGoOlder`/`canGoNewer`), `src/mainview/components/stats/PeriodStepper.tsx`, `src/mainview/components/ProductivityStatsView.tsx` (offset state + relative label), i18n `stats.nav.*`/`stats.rel.*`/`stats.periodPrev.generic`/`stats.momentum.idlePast` (en/ru/es), tip `stats-time-travel`. Tests: `productivityStats.test.ts` (offset window shift, arrow gating, lifetime-anchor invariance, `all` ignores offset), `PeriodStepper.test.tsx`, `ProductivityStatsView.test.tsx` (navigate + hidden-for-all). Plan: `feature-plans/stats-period-navigation.md`.
## 2026-06-29 — Alt/Option-click moves the shell cursor (terminal expert gesture, no chrome)

- **Decision:** Add an iTerm2-style pointer gesture to the live terminal (`TerminalView.tsx`): `Alt`/`Option` + left mouse-down on a cell walks the **shell/readline** cursor to that cell. The shell-vs-TUI decision lives on the **backend** (`tmuxAltClickMoveCursor` RPC + pure logic in `src/bun/tmux-alt-click.ts`): tmux is asked what runs in the clicked pane, and only live shell panes (`zsh/bash/fish/…`), not in copy-mode, with the click on the cursor's row, receive `send-keys Left/Right × N` (target clamped to the row's text so a click right of EOL lands at EOL). Horizontal-only. The renderer keeps a local CSI-arrows fallback for bare non-tmux PTYs. No visible control of any kind — placement is the terminal surface itself; discoverability is a single `score: 4` "Did you know?" tip. No `keymap.ts` entry (mouse gesture, not an app shortcut), no menu/palette/setting.
- **Rationale:** The obvious renderer-side gate — `!term.hasMouseTracking()` — **cannot work**: dev3's tmux runs `mouse on`, which keeps SGR tracking on the outer terminal permanently (verified empirically; decision 098), so the renderer can't tell a shell pane from Claude Code/vim/htop. Delegating to tmux (`pane_current_command`) is the only reliable signal, and it keeps the gesture **inert in mouse-owning TUIs** — better yet, the SGR bridge now forwards the Alt modifier bit, so Claude Code's built-in alt-click finally receives a real M-click through dev3. Vertical motion is deliberately omitted — in a shell Up/Down maps to history, not cursor movement. As an `expert_shortcut` layered on an existing surface it adds **zero** chrome, sidestepping the project's #1 toolbar/inspector button-creep anti-pattern; the keyboard arrows remain the always-available equivalent, satisfying "a gesture is never the only way."
- **Evidence/status:** `Observed` — plan: `feature-plans/alt-click-move-cursor.md`, decision 098. Touches `src/mainview/TerminalView.tsx` (`onAltClickMove` two-path gating + Alt bit in `sgrMouse` + exported `buildCursorMoveSequence` fallback), `src/bun/tmux-alt-click.ts` (pure, unit-tested), `rpc-handlers/tmux-pty.ts`, `shared/types.ts`, `tips.ts` + i18n en/ru/es.

## 2026-06-29 — Instrument & celebrate: countable progress feeds the Velocity Cockpit (standing UX rule)

- **Decision:** Codify a standing product principle (`PRODUCT_UX_BIBLE.md §1.1`, `Proposed`): the read-only Productivity Stats / Velocity Cockpit (`stats`) is a **first-class consumer of every new feature**, not an afterthought. When a feature produces a *countable, repeatable* signal (a thing shipped, a run completed, a streak, a volume, a milestone), the builder should (1) **emit that signal into the stats pipeline at build time** — extend the `getProductivityStats` event shape (`rpc-handlers/productivity-stats.ts`) and/or the pure engine (`utils/productivityStats.ts`) so the data exists even before it is charted — and (2) **surface a visualization on the cockpit when the metric is motivational** (progress/momentum/achievement); diagnostic-only metrics keep the data but stay off the cockpit. Captured as `PRODUCT_UX_BIBLE.md §1.1`, a §10 placement-row, and `ux-architecture.yaml placement_rules.instrument-and-surface-countable-metrics` + the `stats_dashboard.instrumentation` note.
- **Rationale:** The cockpit's whole job is to make shipping *feel* rewarding, and developers respond to a number that ticks up — so the product compounds value if new features feed it by default instead of being instrumented (or never instrumented) much later. **Guardrails prevent this from becoming a bloat license:** the cockpit stays **read-only** (no controls/config/mutation; only the existing time-range switch); a strict **complexity + honesty budget** (prefer one strong signal over many weak ones; a wall of near-zero gauges is worse than none); **forward-only honesty** (an honest "tracking since"/empty-state when a signal is new, never backfilled fake history — the same treatment the LOC views already use); and **numbers must be true** (no inflation/double-counting — a dishonest cockpit destroys the trust that makes it motivating). Diagnostic/operational metrics are explicitly excluded from the cockpit to keep it a motivation surface, not a monitoring console.
- **Status:** `Proposed` (forward-looking practice; the cockpit surface itself is `Observed`, shipped 2026-06-28). Evidence: `PRODUCT_UX_BIBLE.md §1.1` + §10 placement table, `ux-architecture.yaml` (`placement_rules.instrument-and-surface-countable-metrics`, `surfaces.stats_dashboard.instrumentation`), prior cockpit decisions (2026-06-28 ×2), `utils/productivityStats.ts`, `rpc-handlers/productivity-stats.ts`.

## 2026-06-28 — `dev3 remote` backgrounds by default (user-first CLI default; machines opt out with `--no-detach`)

- **Decision:** Bare `dev3 remote` now starts the headless server in the **background** (spawns a detached copy of itself, prints the access link, returns to the shell) — detach is the **default**, not an opt-in `--detach`. Foreground is the opt-out: `--no-detach`. The detached parent forces `--no-detach` on the child it spawns (so the child runs foreground and doesn't recursively detach). All **supervised/machine** callers pass `--no-detach` explicitly: the systemd unit (`buildExecStartArgs` → `dev3 remote start --no-detach`, since `Type=simple` tracks the foreground process), the Docker `CMD`/README run-commands (the container's main process must stay foreground), and the `debug-ui` skill (keeps the server tied to its tmux pane). `--detach` is still accepted as a no-op alias for the default; `--no-detach` wins if both are given.
- **Rationale:** Direct application of the north-star principle (`PRODUCT_UX_BIBLE.md §1.0`): `dev3 remote` is a **hand-typed** command, and a lazy human's most-wanted outcome is "start it and give me my shell back," then manage it with `status`/`url`/`stop`. Docker/nginx/`Type=simple` default to foreground, but a person typing the command is not Docker — so the human gets the lazy default and the **machine** contexts (which don't mind verbosity and actually *need* foreground) pay the one extra flag. This deliberately diverges from the Unix daemon convention because the convention optimizes for supervisors, and here the primary caller is a human. Rejected: keeping foreground default + `--detach` (made the common remote-Linux walk-away case pay the flag — backwards: it taxes the human to please a convention the human isn't bound by).
- **Evidence/status:** `Observed` — `src/cli/commands/remote.ts` (`startRemote` default-detach + `--no-detach` foreground branch; `startDetached` re-spawns `process.argv` minus detach flags, appends `--no-detach`), `src/cli/commands/remote-service.ts` (`buildExecStartArgs` prepends `--no-detach`), `docker/Dockerfile` + `docker/README.md` (CMD/run use `--no-detach`), `.claude/skills/debug-ui/SKILL.md`. Tests: `remote.test.ts` (default-detach + child `--no-detach`; `--no-detach` foreground in-process boot), `remote-service.test.ts` (ExecStart includes `--no-detach`).

## 2026-06-28 — Browser-mode application menu bar (the native menu's stand-in in Remote Access)

- **Decision:** In Remote Access / browser mode (`isElectrobun === false`) the app renders its **own** application menu bar (`AppMenuBar.tsx`) as the **first row of the window, above `GlobalHeader`** — mirroring where the OS draws the native menu bar on desktop. It is built from the **same** `buildApplicationMenu(MenuContext)` source as the native menu — that pure module was **relocated `src/bun/application-menu.ts` → `src/shared/application-menu.ts`** (its only electrobun dependency was a type-only import, erased at build) so both processes share one definition with **no RPC and no drift**. The renderer reuses the `MenuContext` it already computes for `updateMenuContext`, so enable/disable is **local and synchronous**. Items dispatch through the **same `handleMenuAction` router** the native menu and command palette use. Filtering (`buildBrowserMenu`): native-only `role` items (clipboard/window/quit) are dropped (the browser provides them); any action not in `menuRouter`'s exported `BROWSER_HANDLED_ACTIONS` (the bun-only items — new-window, check-for-updates, devtools, zoom, open-logs, remote-QR) is dropped; roadmap (`NOT_YET_IMPLEMENTED`) items are dropped; context-gated items render **greyed** when their task/project/terminal context is absent. `menuRouter` gained browser handlers for `about` (fetches version via `getAppVersion` RPC → `rpc:showAbout`), `hard-refresh`, `gauge-demo`/`viewport-lab`, and the `help-*` external links (`window.open`). Net browser top-levels: **dev-3.0 · File · Task · Project · View · Terminal · Help** (Edit & Window vanish — all-native). Collapses to a single `☰` button below 640px (mobile remote). **Browser-only:** never mounted in Electrobun (native menu owns it there).
- **Rationale:** Remote/browser mode lost the app's *canonical, complete* action surface — the native menu — leaving only the keyboard-only ⇧⌘P palette mirror. A dedicated top strip restores it at desktop parity while adding **zero** pressure to `GlobalHeader`'s dense right cluster (toolbar-button-creep is the #1 project anti-pattern) — it is a fresh surface with its own budget. Sharing one menu definition (relocate, don't duplicate) is the project's single-source-of-truth ethos; gating clickability on real router coverage avoids the native menu's pre-existing "enabled but inert" items (e.g. mark-completed) silently no-op'ing in the browser. Labels stay **English** (the native menu is English-only and not in i18n) so the two menus never diverge — a documented exception, not a scope shortcut. Rejected: merging into `GlobalHeader` (header = location+switching only; would bloat the dense cluster); a `getApplicationMenu` RPC (needless round-trip for data the renderer can build locally); duplicating the menu in the renderer (guaranteed drift); a `☰`-only form on wide screens (loses the menu-bar affordance — `☰` is the narrow-width fallback only).
- **Evidence/status:** `Observed` — `src/shared/application-menu.ts` (relocated, `isComingSoonAction`), `src/mainview/components/AppMenuBar.tsx` (`buildBrowserMenu`), `src/mainview/menuRouter.ts` (`BROWSER_HANDLED_ACTIONS` + browser handlers), `src/mainview/App.tsx` (`menuContext` useMemo, browser-only mount above `GlobalHeader`), i18n `menubar.*` en/ru/es, tip `remote-menu-bar`. Tests: `AppMenuBar.test.tsx`, `menuRouter.browser.test.ts`. Plan: `feature-plans/browser-app-menu-bar.md`.
## 2026-06-28 — Narrow-viewport (mobile) doctrine: carousel/one-at-a-time everywhere

- **Decision:** Promote the board carousel from a one-off feature into a **product-wide doctrine** for sub-768px (mobile / `dev3 remote`) viewports, codified as bible §12 (rewritten) + the `responsive` block in `ux-architecture.yaml`. The single principle: *on a narrow viewport show exactly one sibling at a time and move between siblings by swipe + a visible pager* — applied to columns, tasks-in-column, terminal panes, active tasks, settings sections, and diff files. Key rulings: (1) **Breakpoint reconciliation** — gate **layout** on the reactive `useNarrowViewport(768)` (Tailwind `md`); `useCompact()` (1600) is dense-desktop only; `useMobile()` (1024, mount-once, `screen.width`) drives **only** the `<meta viewport>` choice; never gate layout on `isElectrobun`. (2) **Gesture law** — full-surface swipe is allowed on scroll-body surfaces (board) but **forbidden** on live-content (terminal/diff), which use an explicit pager; every swipe has a button + keyboard equivalent; `prefers-reduced-motion` honoured everywhere. (3) **Per-surface adaptation map** for all §5 surfaces (modals → full-bleed/sheet, inspector 2×2 → summary bar + actions sheet, diff aside → file-picker sheet, settings tabs → one-section carousel, context menu → action sheet, global header → kebab reflow). (4) **Touch nav & action reachability** — the keyboard-only palettes + Cmd+1..9 + hint overlay are dead on touch and the native menu is absent in remote, so the command/action palette gains a mandatory touch entry + responsive width and becomes (with per-object action sheets) the canonical mobile action surface; **no feature may be touch-unreachable**. (5) A single reusable **`BottomSheet`** primitive is mandated (none exists). New anti-patterns: touch-unreachable-feature, fixed-width-overlay-on-narrow, non-wrapping-toolbar-on-narrow, layout-gated-on-transport-not-width.
- **Rationale:** Phone-over-remote is the real secondary form factor, but the app is effectively fixed-width desktop (only ~13 Tailwind responsive prefixes in the whole renderer; nav/inspector/diff/modals/palettes all overflow or are keyboard-only at 390px — verified by a 3-agent code audit). Generalising the proven carousel idiom (one principle, evidence-grounded per surface) avoids a fork into a separate mobile app and keeps every surface a responsive view-mode of its desktop self. The breakpoint reconciliation fixes a real bug: the prior doc said "<1024 / useMobile" while the shipped gate is "768 / useNarrowViewport", and `useMobile` is mount-once so it can't gate live-resizing layout.
- **Status:** `Observed` (board carousel + hover-preview disable) / `Proposed` (everything else — this is the doctrine + implementation brief). Audit evidence: `GlobalHeader.tsx`/`useCompact.ts`, `ProjectView.tsx` (isBrowserMode → `ActiveTasksStrip`), `SplitLayout.tsx`, `PaletteShell.tsx` (34rem), `TaskInfoPanel.tsx`, `TaskDiffViewer.tsx` (22rem aside), `*Modal.tsx`, `useNarrowViewport.ts`/`useMobile.tsx`/`useViewport.ts`, `TerminalView.tsx` (touch→mouse), `ExtraKeyBar.tsx`. Full plan: `feature-plans/narrow-viewport-doctrine.md`.

## 2026-06-28 — Remote Access modal: network-interface (IP) selector, tunnel-off only

- **Decision:** In the **Remote Access modal**, when the Cloudflare tunnel is **off**, render a compact styled native `<select>` **directly above the URL code block** (under the QR + countdown) that lists every candidate IPv4 the machine exposes — each non-internal interface address **plus loopback `127.0.0.1`** — and rebuilds the URL + QR for the chosen host. The selector is **hidden when the tunnel is on** (the public `trycloudflare` URL is authoritative then). Default selection = the backend's existing auto-pick (first non-internal IPv4), so behaviour is unchanged until touched; the chosen host is **preserved across the 25s token refresh** and is **session-local** (not persisted in v1). Backend enumerates via `os.networkInterfaces()`, `getAccessUrl(host?)`/`generateQrDataUrl(host?)` take an optional host, and the requested host is **validated against the enumerated allow-list** (no arbitrary host injection).
- **Rationale:** The previous behaviour silently embedded one auto-chosen interface, which breaks on multi-homed boxes (VPN/utun, Docker bridges, multiple NICs) and offered no way to get a `localhost` URL for the SSH-forward path. This is **configuration inside an existing modal**, not a new surface/nav/toolbar control, so it adds no complexity-budget pressure. Placed adjacent to the URL it controls (reads top-to-bottom: pick address → see URL → optionally flip to tunnel) rather than below the tunnel toggle (too far from its effect). A **styled native `<select>`** reuses the established pattern (`ProjectSettings.tsx:243`), is accessible for free, and renders identically in the Electrobun WKWebView and the headless browser — it is a form control, not a banned native OS dialog. Hidden-when-tunnel-on avoids disabled-control clutter. Session-local (not persisted) keeps v1 simple; localStorage persistence noted as a future enhancement.
- **Evidence/status:** `Planned` — plan: `feature-plans/remote-access-interface-selector.md`. Touches `remote-access-server.ts` (interface enumeration + host-aware URL/QR), `rpc-handlers/remote-access.ts` (`getRemoteAccessQR` host param + interface list), `App.tsx` (modal selector), i18n en/ru/es.

## 2026-06-22 — Feature-discovery tips are surface-aware and distributed beyond the board

- **Decision:** Tips gain a **required** `contexts: TipContext[]` field (`board | terminal | diff | settings | preparing`) and `selectTip(state, context?)` uses it as a **sort boost, never a filter**: when a carrier supplies its surface context, tips matching it lead (still by score tier) and are exhausted first; once the matching pool drains, the remaining tips surface — so a surface never runs dry. A **compact tip card** now renders in the `ActiveTasksSidebar` (under the search box), which is the only tip carrier visible in the **task/terminal view**; the board keeps its `KanbanColumn` `TipCard`. Both carriers share a new `useTipRotation(context, disabled?)` hook (extracted from KanbanBoard's prior inline rotation logic). The board passes context `"board"`, the sidebar `"terminal"`. A new top-priority tip (`terminal-select-copies`, score 5) teaches that selecting text in any terminal pane auto-copies it.
- **Rationale:** The tip engine previously reached **only** the Kanban board (`TipCard` mounted solely in `KanbanColumn`), so the highest-pain discovery facts (terminal copy-on-select, diff review, ports) never reached users living in the terminal. The sidebar is the right anchor because it is mounted in `ProjectView` for the task view and is mutually exclusive with the board in the left split (no double tip card). Context as a **boost, not a filter** was an explicit product call: a terminal tip should *lead* in the terminal but the surface must still cycle the full catalogue, so users keep discovering everything. Rejected: a ticker/marquee (annoying), a permanent terminal footer (chrome creep on the densest full-bleed surface), an exclusive "show only here" flag (unnecessary complexity). The `contexts` field is required (not optional) and all ~99 existing tips were migrated, so coverage can be enforced by test rather than decaying silently.
- **Evidence/status:** `Observed` — `tips.ts` (`TipContext`, `contexts`, context-aware `selectTip`), `hooks/useTipRotation.ts`, `KanbanBoard.tsx`, `ActiveTasksSidebar.tsx`, `TipCard.tsx` (`compact`), i18n en/ru/es, `__tests__/tips.test.ts`. Plan: `feature-plans/terminal-select-to-copy-affordance.md`. The one-time moment-of-action copy toast also shipped: the first successful select-to-copy on a machine fires a `toast.info` (gated by a localStorage flag, not pre-set for existing users) — `TerminalView.tsx` `onMouseUp` success path, key `terminal.copyHint`.

## 2026-06-22 — Active Tasks sidebar: within-group ordering is oldest-first (work-queue, not feed)

- **Decision:** Inside every group of the `ActiveTasksSidebar` (both built-in status groups and custom-column groups), tasks are sorted **oldest-first by `movedAt`** — the longest-waiting task sits at the **top**. The direction is **uniform across all groups** (no per-status flipping). Tasks lacking `movedAt` fall to the bottom of their group, tiebroken by `seq`. The sidebar does **not** reuse the kanban's `sortTasksForColumn` (that couples to the global `taskDropPosition` setting + manual `columnOrder` drag-reorder, neither of which the sidebar exposes).
- **Rationale:** The sidebar is a **work queue, not a feed**. A feed (chat/notifications) is newest-first because you read each item once; a queue must be FIFO-by-urgency, because the oldest item is the most at risk of being forgotten (starvation) — and for the review/questions groups the agent is *blocked waiting on the user*, so the longest wait is the most urgent. This matches the existing **attention/bell scope**, which already sorts oldest-first for the same anti-starvation reason, keeping the surface internally consistent. A single uniform direction beats a semantically-cuter per-group split (review=oldest, in-progress=newest) because direction flipping mid-list breaks scan predictability; the always-visible per-card age badge already conveys absolute freshness regardless of direction. This reverses the naive "freshest on top" intuition by design. Known trade-off: a pure `movedAt` sort can separate variant siblings of the same `groupId` (the kanban clusters them); accepted for now since the sidebar is a flat actionable queue and variant dots still signal membership — revisit with group-clustering if it reads poorly.
- **Evidence/status:** `Observed` — `ActiveTasksSidebar.tsx` (`byMovedAtOldestFirst` comparator applied per group). Builds on the 2026-06-15 status-age-badge decision (`movedAt` = time-in-current-status) and the attention-scope oldest-first sort.

## 2026-06-21 — Hint navigation is a cross-surface primitive; keyboard-first expert layer

- **Decision:** Promote the board-only Vimium hint (PR #704) to a **surface-agnostic** `HintOverlay` that scans `[data-hint-id]` (placed only on the innermost clickable element) and commits via `element.click()`. Targets: task cards, dashboard project rows + attention tasks, sidebar task rows. **Policy: a hint maps to a navigation/open destination only — never a mutation or destructive action** (settings/remove/reorder are not hinted), mirroring the palette's destructive-exclusion rule. Activation is **data-driven** (`querySelector("[data-hint-id]")`) rather than gated to one screen. Hotkey: keep bare **`F`** (Vimium standard) + add a Mac-friendly **`⌘G`** alias. Added a Linear/GitHub **`g`-prefix go-to** layer (`g` then `d`/`p`/`t`/`s` → dashboard/project/tasks/settings), **`/`** to focus search, and **`c`** as a bare-key alias for new task. All bare/sequence keys are matched on **`e.code`** (physical key) so they work on Cyrillic/Hebrew/any layout, and are gated by `isTypingContext()`.
- **Rationale:** Generalizing the existing overlay (not a per-surface clone) avoids the "new component for old pattern" anti-pattern; the navigation-only policy avoids "row action explosion"/"dashboard junk drawer". The layout bug was the real defect — both activation and hint typing read `e.key`, so the whole feature was Latin-layout-only; `e.code` fixes it. `f` is kept because it is the world-standard hint key and `⌘G` gives a clean published chord; bare `g` is reserved for the go-to prefix (so hints must not consume it).
- **Evidence/status:** `Observed` — `HintOverlay.tsx`, `utils/hintLabels.ts` (`codeToHintChar`), `App.tsx` (`useGlobalShortcut` chain, `GO_TO_TARGETS`, `findVisibleSearchInput`), `ActivityOverview.tsx`, `ActiveTasksSidebar.tsx`, `TaskCard.tsx`, `LabelFilterBar.tsx`, `keymap.ts`, i18n en/ru/es, tips (`task-hint-nav`, `keyboard-go-to`). Plan: `feature-plans/hint-navigation-generalization.md`. Decision record 076.

## 2026-06-19 — Keyboard-shortcut registry as single source of truth + a unified two-tab reference overlay

- **Decision:** Establish `src/mainview/keymap.ts` as the **single source of truth** for every
  app-level keyboard shortcut (declared as data: `id`, per-platform `keys`, `descKey`, `category`).
  The in-app reference is **one** `KeyboardShortcutsModal` (Modal surface, mirroring
  `TmuxCheatSheetModal`'s shell) with **two tabs — App | Terminal (tmux)**: the App tab renders from
  `keymap.ts`; the Terminal tab folds in the existing tmux cheat-sheet content. Entry points:
  (1) native **Help → Keyboard Shortcuts** — wires the currently **dead** `help-keyboard-shortcuts`
  menu action through `menuRouter` (Help → Tmux Cheat Sheet opens the same modal on the Terminal tab);
  (2) keyboard **⌘/ (Ctrl+/ on Linux)**, owned by the `App.tsx` keydown handler in capture phase
  like the palettes, label shows `(⌘/)`; (3) the **⇧⌘P command palette** via a `commands.ts` entry.
  **No** toolbar/header/breadcrumb button. The same registry data also drives the README table and
  the website (`docs/index.html`) section — the legitimate standalone "page" form.
- **Registry is documentation + test, NOT a dispatcher (chosen):** the giant `App.tsx`
  `useGlobalShortcut` if-else stays the executor; refactoring it into a registry-driven dispatch was
  rejected as a risky rewrite of central, edge-case-heavy code (capture phase, terminal focus, zoom,
  `e.code` vs `e.key`). A vitest test guards drift (every spec valid + unique; best-effort flag of
  App.tsx handlers missing from the registry).
- **Placement rationale:** the overlay is `onboarding/help` + `expert_shortcut`, the same class as
  the tmux cheat sheet — it belongs in the Help menu + a keyboard chord + the palette, never on a
  toolbar (toolbar-button-creep is the #1 anti-pattern) and never as a navigation destination
  (ephemeral reference ≠ a place; the nav budget is ≤7). One modal with tabs (vs two sibling modals)
  per the user's "единый вид" requirement; the tab split keeps ~40 tmux bindings from drowning ~25
  app shortcuts. `⌘/` over bare `?`: the live terminal must still receive a bare `?`.
- **Scope:** Follow-up implementation — `keymap.ts`, `KeyboardShortcutsModal` (App + Terminal tabs),
  ⌘/ handler, `menuRouter` `help-keyboard-shortcuts` case, `commands.ts` palette entry, menu label,
  i18n en/ru/es, tests, plus `AGENTS.md`/README/website + the CLAUDE.md registry rule. The
  `keyboard-shortcuts` tip already exists (`tips.ts`).
- **Status:** `Proposed` (design locked via interview 2026-06-19; not yet implemented). `/ux-principal`
  consulted (manifest read; placement/source-of-truth/entry points decided before coding). See

## 2026-06-03 — Narrow-viewport carousel navigation (mobile / remote), planning

## 2026-06-19 — Diff review is a 3-day persisted safety net + explicit "Reset review"; document the diff viewer surface

- **Decision:** The inline diff-review in `TaskDiffViewer` is persisted per task in `localStorage` so it survives unmount, diff reload, and app restart — but as a **short-lived safety net, not a permanent store**: a **3-day TTL** (from when the review was first created) auto-expires it on next read. The clipboard becomes pure *transport*, not the store. A new **`Reset review`** control is added to the Review-export card (left Files aside), placed **below** the existing full-width `Copy review` primary. `Reset review` is role `destructive`, low-emphasis ghost-danger styling (`text-danger` + `border-danger/30` + `hover:bg-danger/10`), visible only when ≥ 1 comment exists, and gated behind a `confirm()` dialog (React confirm service — no native dialog). The review is cleared by **Reset** or by **TTL expiry**. Consequence: leaving the surface no longer discards anything, so the old "discard review?" confirm-on-close guard is dropped. As part of this, the previously-undocumented **diff review viewer** surface was added to the manifest (Bible §5.2 + `ux-architecture.yaml: surfaces.diff_review_viewer`).
- **Rationale:** The fatal failure was that the review lived only in volatile clipboard + in-memory React state (`inlineComments` useState, wiped on reload/unmount); an accidental terminal text-selection clobbered the clipboard and the whole review (often 10+ comments over a 1000-line diff) was lost irrecoverably. The real need is narrow — let the user reopen the diff and re-copy after a clipboard mishap — so a permanent "until Reset" store was over-scoped: it would accumulate stale reviews forever. A 3-day TTL covers the realistic re-copy window (the user estimated 1–3 days, capped at 3) while self-cleaning. Reset is destructive (matches the taxonomy's "reset terminal / hard refresh" precedent) so it must never wear primary/accent — that would compete with Copy and invite an accidental data-loss click. No new destination, no inspector toolbar-creep: the whole review lifecycle stays inside the diff viewer.
- **Storage choice:** Kept in `localStorage` (not a `~/.dev3.0/` JSON file). The quota concern is mitigated by a `pruneExpiredReviews` **global sweep** on every diff-viewer mount that drops expired/corrupt entries across all tasks — so the working set stays "reviews touched in the last 3 days" regardless of which tasks are reopened, and orphaned entries (deleted tasks) self-clean. A backend file store was rejected as over-scoped for ephemeral review drafts (new RPC plumbing + async save-per-keystroke + touching the hard-invariant `~/.dev3.0/` zone).
- **Status:** `Observed` (implemented: `TaskDiffViewer.tsx` — `readStoredReview`/`writeStoredReview`/`pruneExpiredReviews` keyed `dev3-inline-diff-review-v1:<taskId>`, restore-on-load, mount-time sweep, `handleResetReview`, navigation/discard guard dropped; keys `infoPanel.diffReviewReset`/`diffReviewResetConfirmTitle`/`diffReviewResetConfirmMessage` in en/ru/es; tip `diff-review-persists`; plan `docs/ux/feature-plans/persist-diff-review.md`).
