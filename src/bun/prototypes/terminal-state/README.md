# Terminal-state snapshot spike

This isolated spike answers whether dev3 can ingest raw PTY bytes headlessly,
retain reconnectable terminal state, and reconstruct the same semantic screen in
a fresh renderer. It is not imported by the app or CLI, does not persist data,
and does not touch tmux sessions or the production terminal path.

## Result

Use the same Ghostty VT core as the renderer. For this spike, retain an ordered
journal of output bytes and resize events, replay it through a new Ghostty core,
and compare semantic cells. Do not ship that journal as a production snapshot:
it grows with session lifetime and replay time grows with total historical
output. Production work should first add or obtain a bounded Ghostty-native
export/import API.

The rejected compact alternative, xterm headless plus its serialize addon,
replayed most fixtures correctly but changed resize history when its ANSI
serialization was consumed by Ghostty. At 10×3 → 6×4 → 12×3, xterm reconstructed
`1234567890AB` / `C  D`; Ghostty's live state was `1234567890AD` / `C`. A compact
snapshot from a different emulator therefore cannot be considered renderer
compatible even when ordinary fixtures match.

## Parser evaluation

| Candidate | Existing renderer compatibility | Alternate screen | Unicode | Snapshot access | Maintenance | Bundle/runtime cost | Result |
|---|---|---|---|---|---|---|---|
| `ghostty-web@0.4.0` raw core + event journal | Exact parser and resize semantics; verified against a fresh Ghostty core | Native support, including hidden primary buffer | Same grapheme and width rules as the renderer | Rich read access but no state export/import; journal is the temporary replay format | Already owned by the renderer; raw API still needs lifecycle validation | No added package. Existing package is 2.14 MiB installed, including a 413 KiB WASM file | Selected for the spike; blocked from production by unbounded state |
| `@xterm/headless@6.0.0` + `addon-serialize@0.14.0` + `addon-unicode-graphemes@0.4.0` | Ordinary cells match, but resize-history semantics diverge when replayed into Ghostty | Supported and serializable | Grapheme addon passed the fixture but is documented as experimental | Compact ANSI serialization with scrollback and common modes; cursor presentation needs side metadata | Mature upstream, but adds a second emulator and Unicode policy | 2.65 MiB unpacked across packages; about 251 KiB runtime JS in the tested install | Rejected because cross-emulator replay is not semantically exact |
| Direct `libghostty-vt` export/import | Same terminal engine, so it is the strongest compatibility path | Native support | Native Ghostty rules | No stable bounded export/import surface is exposed by the current JS package | Upstream API signatures are still evolving; requires a small upstream/native seam | Packaging cost unknown; could reuse the existing WASM/native core | Recommended follow-up, not implementable in this spike |
| Custom VT state machine | Compatibility would have to be rediscovered sequence by sequence | Must be implemented | Must own width tables, grapheme clustering, and upgrades | Full control over a compact format | Highest long-term test and maintenance burden | No third-party bundle, but substantial new code and state | Rejected |

The xterm package sizes are registry `dist.unpackedSize` values. The runtime JS
number is the sum of the installed headless, serialize, and grapheme entry
artifacts measured during the experiment; the dependencies were removed after
the candidate failed resize equivalence.

## Snapshot v1

`TerminalSnapshotV1` is deliberately explicit and fail-closed:

```text
format   = "dev3-terminal-state-spike"
version  = 1
strategy = "event-journal"
parser   = "ghostty-web@0.4.0"
initial  = { cols, rows, scrollback }
events   = output(utf8 | base64) | resize(cols, rows)
```

Binary PTY chunks remain base64 rather than passing through a lossy text
conversion. UTF-8 fixture strings stay readable. Unknown formats, versions,
strategies, and parser identities are rejected. Version 1 is experiment data,
not a persistence promise; a bounded native snapshot must use a new version and
define compatibility before it can enter user state.

The spike asserts that the installed `ghostty-web` package is exactly 0.4.0
before creating a core. The repository may retain its production-compatible
semver range, but a lockfile upgrade cannot silently create state under a false
parser identity; the isolated spike fails and requires an explicit format
review.

The semantic comparison covers the active buffer, hidden primary buffer,
dimensions, cursor position and presentation, title, relevant modes, visible
cells, resolved colors and attributes, wrap markers, and scrollback. Device
status responses generated by Ghostty are returned to a PTY during real capture
so applications do not stall or render an error state.

## Fixtures

| Fixture | Coverage |
|---|---|
| `active-screen.json` | Active screen, title, cursor position |
| `alternate-screen.json` | Alternate screen and restoration of the hidden primary buffer |
| `cursor-modes.json` | Cursor visibility/style/blink and terminal modes |
| `scrollback.json` | Bounded scrollback |
| `colors.json` | Truecolor, palette resolution, bold, underline |
| `unicode.json` | Wide glyph, combining mark, emoji, cell widths |
| `line-wrapping.json` | Soft wrapping and wrap markers |
| `resize-history.json` | Shrink/grow history and post-resize output |
| `real-nvim.json` | Captured Neovim 0.12.4 full-screen PTY stream |
| `real-powershell.json` | Captured Windows PowerShell 5.1 ConPTY stream, title, colors, history, and cursor presentation |

Every fixture is replayed into both a fresh headless state holder and a fresh
Ghostty renderer probe. The latter compares state at detach and again after any
post-reconnect events.

## Real captures

`real-nvim.json` was captured from a real Bun PTY at 80×24 using:

```bash
bun src/bun/prototypes/terminal-state/capture-pty.ts \
  --terminal-responses 700 80 24 /tmp nvim --clean --cmd 'set noswapfile'
```

Raw PTY capture is the default and does not load Ghostty. The
`--terminal-responses` flag is reserved for applications such as Neovim that
issue DSR queries and need emulator replies while rendering their initial frame.
On Windows, Bun 1.3.14 returned a negative Ghostty WASM allocation pointer during
live ingestion even though the same parser passed under Node; keeping the parser
out of raw capture avoids coupling PTY collection to that runtime-specific bug.

`real-powershell.json` was captured from Windows x64 at 100×30 with Bun 1.3.14
and Windows PowerShell 5.1.19041.6456 Desktop using:

```powershell
$psExe = (Get-Process -Id $PID).Path
bun src/bun/prototypes/terminal-state/capture-pty.ts 800 100 30 "$PWD" `
  "$psExe" -NoLogo -NoProfile -File `
  ".\src\bun\prototypes\terminal-state\powershell-probe.ps1"
```

Windows PowerShell 5.1 decoded the UTF-8-without-BOM script literals through its
legacy source encoding, so the captured Unicode labels are mojibake. The fixture
is retained byte-for-byte because it still validates real ConPTY output, ANSI
colors, scrollback, title, cursor presentation, and reconnect replay. The
synthetic `unicode.json` stream remains the Unicode fidelity fixture.

## Known gaps and removal

- Journal size and replay cost are unbounded over a long session.
- The probe isolates each raw Ghostty handle in its own WASM instance after a
  shared-instance experiment corrupted grapheme reads during create/free churn.
  That observation needs a minimal upstream reproduction before designing a
  production host; it also inflates the recorded per-client memory cost.
- The metadata tracker covers title and cursor presentation sequences used by
  the fixtures, not every OSC or private mode.
- Hyperlink targets, shell integration metadata, images, sixel, input ordering,
  backpressure, checksums, compression, and snapshot privacy are not designed.
- The real captures cover macOS Neovim and Windows PowerShell 5.1 ConPTY, but not
  Linux PTYs, PowerShell 7, or a full-screen Windows TUI.

Removal requires deleting `src/bun/prototypes/terminal-state/` and the two
terminal-state package scripts. There is no daemon, migration, stored state,
production import, or tmux cleanup to undo.
