# 163 — ECharts in the artifact starter via SRI-pinned CDN script

## Context

The v1 artifact starter drew its charts with ~150 lines of hand-rolled SVG math (decision 153), which capped both visual quality and the chart types agents could realistically produce. The user asked for a proper charting library so artifacts get beautiful interactive charts with minimal authoring code. Decision 153 rejected chart libraries only in their unpinned runtime-CDN form.

## Investigation

Apache ECharts 6.1.0 won the library choice: declarative option objects (AI-friendly), an SVG renderer (the decision-153 print contract — canvas blurs or vanishes in PDF), rich chart types, and beautiful defaults. Chart.js is canvas-only, ApexCharts covers far fewer types, Plotly is 3-4x heavier with dated defaults. The first iteration inlined the whole minified build into `index.html` as one ~1.1 MB line; that shipped and worked (CSP-safe, offline), but made artifact HTML effectively unreadable for any LLM or agent tooling — a single megabyte-long line breaks naive file reads, diffs, and chat uploads, which is unacceptable for an AI-first product. Note: the cdnjs API's published SRI for `echarts.min.js` 6.1.0 does **not** match the bytes the CDN actually serves; the hash was computed from the served bytes and verified in a browser (API hash gets rejected, computed hash loads).

## Decision

The starter references ECharts via `<script data-dev3-vendor="echarts@6.1.0" src="https://cdnjs.cloudflare.com/ajax/libs/echarts/6.1.0/echarts.min.js" integrity="sha512-…" crossorigin="anonymous" referrerpolicy="no-referrer">`. SRI makes a tampered CDN payload fail closed. Offline, the `dev3Chart` bridge renders a `.chart-unavailable` notice per chart and the rest of the report stays interactive.

The artifact viewer CSP (`src/mainview/utils/artifactDocument.ts`) is **network-open by explicit user decision**: scripts, styles, fonts, images, and media may load from any `https:`/`http:`/`data:`/`blob:` source, and `connect-src` permits `fetch`/WebSocket to any server — artifacts are an integration surface (agents may wire reports to the user's own services or the dev3 dev server), and a sealed network made that impossible. A cdnjs-only allowlist was the interim state and was rejected as too paranoid. The isolation boundary is the iframe sandbox itself (`sandbox="allow-scripts"`, opaque origin): the artifact cannot reach the dev3 app, its files, cookies, or parent page, and servers see requests from a `null` origin (CORS applies). Only legacy vectors stay closed (`object-src 'none'`, `base-uri 'none'`). The bridge (`dev3Chart(element, optionFactory)`) themes charts from the `--dev3-*` tokens via `echarts.registerTheme`, forces the SVG renderer, enables aria descriptions, observes container resizes, and re-renders compact heights through `beforeprint`/`afterprint` (`html.printing`). `artifact-template.test.ts` pins the CDN URL, the presence of `integrity`/`crossorigin`, the offline notice, a ≤120 KB template size (guards against re-inlining), and that the starter itself stays self-contained beyond the ECharts tag; `artifactDocument.test.ts` pins the open CSP.

Two integration subtleties, learned the hard way. (1) The host theme message can be dispatched while the document is still parsing: the shell listener in `<head>` flips `data-theme` before the bridge exists, so charts froze on light-token colors inside a dark page (reported from the real WKWebView viewer). The bridge therefore rethemes from a `MutationObserver` on the `data-theme` attribute — the attribute is the single source of truth, immune to message timing. (2) `echarts.registerTheme` cannot style the HTML tooltip overlay; tooltip colors ride in a base option that `setOption(factory())` merges into on every remount.

## Risks

Artifacts with charts require network access to cdnjs (offline/firewalled machines see the notice, not charts) — accepted trade-off for LLM-readable artifact HTML. cdnjs re-serving different bytes for 6.1.0 would brick charts via SRI rejection (correct fail-closed behavior; bump = new URL + new hash + test string). With the network-open CSP an artifact can exfiltrate its own content to any server — accepted by explicit user decision: artifacts are authored by the user's own agents in the user's own session, so the content is already theirs; the sandbox still prevents the artifact from reading anything beyond itself. Artifacts referencing external resources render only while those origins are reachable.

## Alternatives considered

Full inline vendoring — shipped first, rejected after real use: unreadable artifact HTML for LLMs. Compose-time inlining from the app bundle (template keeps a `src="echarts.min.js"` reference, viewer/download paths splice the app-bundled copy in) — preserves offline and readability but needs pipeline changes across viewer, download, and ZIP paths; deferred as a follow-up if offline charts matter. Chart.js / ApexCharts / Plotly — rejected per Investigation. Custom slim ECharts build — rejected: loses the exotic chart types that make AI reports shine, and size no longer matters once the library is not inlined.
