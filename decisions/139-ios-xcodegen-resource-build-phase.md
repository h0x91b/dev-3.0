# Context

The iOS project is generated by XcodeGen, and its app resources include the asset catalog, bundled Nerd Font, and license files. A successful generic simulator build did not prove these files were copied into the resulting app bundle.

# Investigation

Declaring `resources` as a top-level target key was silently ignored by the XcodeGen version used on this machine. The generated project built without errors but omitted `Assets.car`, app icons, fonts, and licenses from the app bundle.

# Decision

Declare `App/Resources` as a second target `sources` entry with `buildPhase: resources` in `ios/project.yml`. Branding tests inspect this declaration, while simulator builds and asset-catalog inspection verify the packaged output.

# Risks

Future XcodeGen schema changes could alter source-entry behavior, so generated-project drift must remain visible in CI. Resource presence should be verified from the built app whenever this declaration changes.

# Alternatives considered

We rejected committing the generated Xcode project because `project.yml` is the repository source of truth. We also rejected a custom copy script because Xcode's resource build phase correctly compiles asset catalogs and handles packaged fonts.
